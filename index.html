What's the difference?

Beads models a dependency-tracking issue tracker: work items move blocked → ready → done, and a fixed pool of workers pulls the next ready item off one shared queue. Parallelism is capped by how many workers you staff — it's a pull model.

Actors models each task as an independent actor that reacts the instant every dependency's "done" message has arrived, then broadcasts its own completion. There's no central queue: parallelism falls directly out of the dependency graph — it's a push/reactive model. Cap its concurrency above to compare it against Beads with the same resource budget.

Dynamic DAGs: pick the "Dynamic: flaky pipeline" preset to see a node decide what happens next based on its own result — Run tests "sees" pass or fail and spawns a different set of downstream nodes accordingly. The seed above determines that decision deterministically per node id, so Beads and Actors always face the same dynamic choices even though they discover and execute nodes in a different order — the comparison stays fair. Reroll the seed to see a different draw.

Jobs vs. agents: a node can be marked kind: "agent" to model an LLM call instead of a deterministic job. Agent nodes share one concurrency limit (a stand-in for an API rate limit) that's separate from the worker-pool/actor-cap settings above — deterministic job nodes are never throttled by it. Try "Mixed: jobs + LLM agents": Actors would normally fire all 4 agent calls at once, but with the limit set to 2, two of them queue behind the rate limit even though nothing else is blocking them — while the independent format job runs immediately, unaffected.
